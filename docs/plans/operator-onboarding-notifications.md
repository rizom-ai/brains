# Plan: Operator Onboarding Notifications

## Status

Proposed. Depends on `docs/plans/passkey-setup-bootstrap.md` exposing first-passkey setup state through an internal contract.

## Context

Most current chat-interface behavior is request/response: a user sends a message, and the interface replies in that conversation. First-passkey setup is different. It is a system-initiated operator notification: the brain needs to push a sensitive onboarding message to trusted operators.

This plan introduces that push-notification path separately from the OAuth/passkey bootstrap itself.

## Goal

Add a generic operator notification path that can deliver sensitive setup/onboarding messages through operator-facing interfaces, starting with Discord DMs to configured anchors.

## Non-goals

- Do not generate or validate setup tokens here; auth-service owns that.
- Do not make Discord responsible for auth setup state.
- Do not send setup URLs to public channels.
- Do not infer secret delivery from `allowedChannels`.
- Do not require GitHub Actions logs, hosted log scraping, or doc-brain deployment.

## Notification contract

Introduce a system notification/request with a shape like:

```ts
interface OperatorNotification {
  audience: "anchors";
  sensitivity: "secret" | "normal";
  title: string;
  body: string;
  actionUrl?: string;
  dedupeKey?: string;
}
```

Exact naming can follow existing messaging conventions. The key design point is that this is a shell/plugin notification contract, not an auth-service-to-Discord call.

## Responsibility split

```text
AuthService
  owns setup token/state and exposes "operator setup required"

AuthServicePlugin
  bridges setup-required state into an operator notification request

Operator notification layer / message bus
  defines audience and delivery semantics

DiscordInterface
  owns Discord transport and sends Discord DMs
```

## Anchor recipient resolution

Discord delivery should use configured anchors as trusted operators:

- Resolve configured `discord:*` anchors.
- DM the same setup URL to each Discord anchor.
- First successful passkey registration wins.
- Later clicks see setup already completed or invalid token.

If the current permission service can check anchor status but cannot list anchors, add a narrow read API such as:

```ts
context.permissions.listAnchors("discord"); // -> string[]
```

or an equivalent shell-level recipient resolver. Discord should not scrape raw app config if a cleaner permission/recipient API is available.

## Push delivery behavior

- Discord queues the notification until its client is connected.
- Immediately before sending a secret setup URL, the integration should avoid sending if setup has already completed or the URL is no longer valid.
- Delivery failures should be logged without including the setup URL.
- No public/channel fallback by default.

Optional future fallback: an explicit private operator channel config may be added if DM delivery proves unreliable for teams.

## Discord setup message

Suggested copy:

```text
Passkey setup required: https://domain/setup?token=...

This setup link is single-use. The first configured anchor to register a passkey completes operator setup and closes the link.
```

## Docs

Update onboarding docs to state:

- The setup token is generated by Brain, not provided as env.
- Hosted setup receives the generated URL through configured operator notification.
- Initial Discord delivery DMs configured `discord:*` anchors.
- The same URL may be sent to multiple trusted anchors; it is single-use and first registration wins.
- Logs are local/dev fallback only.

## Tests

Add focused tests for:

- auth-service-plugin emits/requests operator notification when setup is required
- no notification is requested when a passkey already exists
- Discord resolves `discord:*` anchors and sends DMs
- Discord queues or safely handles notification before client readiness
- Discord delivery failure logs do not include the setup URL

## Validation

- Run targeted auth-service tests for the bridge.
- Run Discord/interface typecheck and tests.
- Run Relay typecheck if Relay docs/config are touched.
- Run env schema/bundled model tests only if schema artifacts change.
- Add changesets for affected packages when implemented.
