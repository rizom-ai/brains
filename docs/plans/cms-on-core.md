# Plan: CMS on core (no webserver, no site-builder)

## Status

Proposed.

## Context

CMS = Sveltia CMS — a client-side SPA with a GitHub API backend. It needs two static files: `admin/config.yml` (schema-driven field definitions) and `admin/index.html` (CDN bootstrap).

Today those files are generated by `plugins/site-builder/src/lib/cms-config.ts` and written into the static-site build output. The webserver plugin then serves them. The `core` rover preset deliberately excludes both site-builder and webserver, so CMS is unavailable on core today.

But the actual coupling between CMS and the site is shallow:

- CMS backend is GitHub API, not the brain
- Sveltia loads from a CDN — no bundling
- The brain doesn't see CMS traffic — it just sees the resulting git commits via directory-sync
- Core already has directory-sync + an MCP HTTP transport, the two prerequisites

The right architecture pulls the config out of the build output entirely and serves it from a brain endpoint. A thin hosted Sveltia page reads the config from any brain URL and runs Sveltia against it. Brain side: zero files added to the content repo, no webserver needed.

## Goal

Operators on `preset: core` can use Sveltia CMS to edit their content repo by:

1. Brain exposes a read-only `cms-config` endpoint on the existing MCP HTTP transport
2. Operator visits a hosted Sveltia page (rizom-managed static asset) with `?brain=<url>`
3. Hosted page fetches the config, hands it to Sveltia, Sveltia talks to GitHub
4. Edits flow through GitHub → directory-sync → entity updates

**No files written to the content repo. No webserver on the brain. No new network surface beyond the existing MCP HTTP.**

## Non-goals

- Hosting Sveltia from the brain
- OAuth proxying or auth on the brain side (Sveltia handles GitHub auth client-side)
- Replacing site-builder's CMS generation for the existing site-builder path (it can keep working from the same shared module)
- Solving auth/access-control for the config endpoint in v1 — config is not secret

## Design

### Extract the CMS-config generator into a shared module

Move the schema → Sveltia config logic out of `plugins/site-builder/src/lib/cms-config.ts` into a shared module:

- New: `shared/cms-config/`
  - `generateCmsConfig(options): CmsConfig` — produces the YAML structure
  - Hosted-page bootstrap (HTML) is _not_ generated by the brain anymore — the rizom-hosted Sveltia page is its own static asset

The generator takes entity registry callbacks and returns a config object — no site-builder coupling, no filesystem writes.

### New `cms-config` resource on the brain

Add a single read-only resource served on the existing MCP HTTP transport:

- Path: `cms-config` (resource URI)
- Response: serialized CMS config (YAML or JSON — Sveltia accepts both)
- Builds on demand from the entity registry, so it's always current
- No auth in v1; the config is not sensitive

This is a ServicePlugin or core-level integration — implementation detail. The key is it lives on the existing MCP HTTP server, so core gets the endpoint for free.

### Hosted Sveltia page (separate deployment)

A static page at e.g. `cms.rizom.ai`:

```
1. Reads ?brain=<url> from query string
2. Fetches <url>/cms-config (or the resource path)
3. Initializes Sveltia with that config
4. Sveltia's GitHub flow takes over from there
```

This is ~50 lines of HTML/JS. Deployed once, used by every brain instance. Lives outside this monorepo (or in a `tools/cms-host/` directory).

### Site-builder keeps its current behavior

Site-builder continues to generate `admin/*` into the build output for deployments that have a webserver. It just imports the shared module instead of owning the generator. No coordination with the new endpoint.

### Rover preset wiring

`core`, `default`, and `full` presets all get the cms-config resource (it lives on MCP HTTP, which all three presets have). For deployments with a webserver, the existing in-build CMS files keep working — operators can use either entry point.

## Including base (note) entities

### Current state

Notes (`entityType: "base"`) have a frontmatter schema and aren't excluded from `generateCmsConfig`, so they're nominally part of the CMS today. But existing notes that contain bare `---` in the body (used as a markdown horizontal rule, section separator, or quoted content) break Sveltia's frontmatter parser — the first `---` is interpreted as a frontmatter delimiter, and Sveltia either fails to load the file or corrupts it on save.

This is why notes have been treated as "excluded in practice" — they're configured in but unusable.

### Fix: configure base-note collection as markdown-only in Sveltia

Don't touch any note files. Don't introduce a normalization step. Configure the base/note collection in the generated CMS config to use Sveltia's `markdown` format instead of `frontmatter`:

- Sveltia loads the entire file as a markdown body widget
- No `---` parsing — bare horizontal rules and section dividers are preserved verbatim
- User edits markdown directly; commit goes back as-is
- Title is **not** a separate form field — the H1 in the body is the title

The brain already handles title extraction from H1 when no frontmatter is present (`entities/note/src/adapters/note-adapter.ts:85-94` — fallback chain: frontmatter title → H1 → null). No brain-side changes needed; the entire fix lives in `generateCmsConfig`.

### What changes in the generator

- `generateCmsConfig` produces a different collection shape for `entityType === "base"`:
  - `format: "markdown"` (not `"frontmatter"`)
  - Single `body` field (no separate frontmatter widgets)
  - `folder: "."`, `extension: "md"`, `create: true` — same as today
- All other entity types (typed: blog, post, link, etc.) keep the current frontmatter-based shape — unchanged

### Tradeoffs

- **Title as H1 only**: editing the title means editing the first heading in the body. Acceptable; H1-as-title is a normal markdown convention. If a note has no H1, the title falls back to filename (already the brain's behavior).
- **No frontmatter widgets for notes**: tags, custom metadata fields aren't editable via CMS forms for base notes. If we ever want them, we add them via the typed entity types where the frontmatter contract is already explicit.
- **Notes with frontmatter still work**: brain's adapter still parses frontmatter when present. Round-trip preserves it. The CMS just doesn't render it as form fields for base notes.
- **Zero migration**: existing notes work as-is, regardless of whether they have frontmatter or bare `---` in body.

## Operator flow on core

```
# Steady state (no setup, no migration):
# 1. Brain runs preset: core (MCP HTTP enabled, no webserver)
# 2. Operator visits https://cms.rizom.ai/?brain=https://my-brain.example.com
# 3. Hosted page fetches the brain's cms-config
# 4. Sveltia renders, operator authenticates with GitHub
# 5. Edits commit to the content repo
# 6. directory-sync pulls them in, brain reflects the changes
```

## Open questions

1. **Where does the cms-config resource live?** Shell-level (always available), or as a small plugin enabled per preset? Shell-level is simpler; plugin gives opt-out.
2. **Does the hosted Sveltia page live in this monorepo?** A `tools/cms-host/` directory keeps it together; a separate repo lets it deploy independently. Probably the latter.
3. **Auth on the brain endpoint?** Config is not secret (it's just schema metadata), but exposing the entity-type list publicly is a minor information disclosure. Acceptable for v1; revisit if it becomes a concern.
4. **Migration UX for the normalize-frontmatter command** — should it create a single commit, or one per file? Single commit keeps history clean.

## Verification

1. `shared/cms-config` builds and tests pass
2. Schema → CMS config output is byte-identical between the old site-builder path and the shared module
3. Brain on `preset: core` exposes the cms-config resource on MCP HTTP; curling it returns the expected YAML
4. Hosted Sveltia page successfully boots against a core-preset brain
5. CRUD against base notes through Sveltia round-trips correctly, with body content containing `---` preserved verbatim
6. CRUD against typed entities (blog, link, etc.) still works through the frontmatter-format collections
7. `default`/`full` presets still serve `admin/*` from build output unchanged

## Risks

- **Schema drift mid-session**: the config resource builds on demand, so it's always current. Not a risk.
- **Hosted-page dependency**: requires rizom to deploy and maintain `cms.rizom.ai`. If the page goes down, no operator can use the CMS until it's back. Mitigation: page is ~50 lines of static HTML/JS, easy to self-host as a fallback.
- **Title editing UX for notes**: title-as-H1 means renaming a note means editing its first heading. Some users may expect a separate title field. Mitigation: documentation; also, the typed entity types (blog, post, etc.) still have proper title fields for cases where structured metadata matters.
- **Sveltia GitHub OAuth setup**: hosted page needs a registered OAuth app. One-time setup per CMS deployment.
