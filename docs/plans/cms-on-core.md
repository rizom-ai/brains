# Plan: CMS on core (no webserver, no site-builder)

## Status

Proposed.

## Context

CMS = Sveltia CMS — a client-side SPA with a GitHub API backend. It needs two static files: `admin/config.yml` (schema-driven field definitions) and `admin/index.html` (CDN bootstrap).

Today those files are generated by `plugins/site-builder/src/lib/cms-config.ts` and written into the static-site build output. The webserver plugin then serves them. The `core` rover preset deliberately excludes both site-builder and webserver, so CMS is unavailable on core today.

But the actual coupling between CMS and the site is shallow:

- CMS backend is GitHub API, not the brain
- Sveltia loads from a CDN — no bundling
- The brain doesn't see CMS traffic — it just sees the resulting git commits via directory-sync
- Core already has directory-sync, which is the one prerequisite

Decoupling the CMS file generation from site-builder lets core get CMS for free, without adding a webserver.

## Goal

Operators on `preset: core` can use Sveltia CMS to edit their content repo, with no webserver and no site-builder, by:

1. Their brain auto-maintains `admin/config.yml` + `admin/index.html` inside their content repo
2. Directory-sync pushes those files to the git remote
3. The operator opens a hosted Sveltia instance (CDN-served HTML) pointed at their repo's `admin/` path
4. Edits flow back through GitHub → directory-sync → entity updates

No new public network surface on the brain. No webserver in the runtime. No coupling to site-builder.

## Non-goals

- Hosting Sveltia from the brain (the CDN-served bootstrap is fine)
- Adding any HTTP endpoint to core
- OAuth proxying or auth on the brain side (Sveltia handles GitHub auth client-side)
- Replacing site-builder's CMS generation for the existing site-builder path — that keeps working

## Design

### Extract the CMS-config generator into a shared module

Move the schema → Sveltia config logic out of `plugins/site-builder/src/lib/cms-config.ts` into a shared module that both site-builder and a new CMS plugin can import:

- New: `shared/cms-config/` (or fold into `shared/utils/`)
  - `generateCmsConfig(entityRegistry): string` — produces the YAML
  - `generateCmsBootstrap(): string` — produces the static index.html

The generator depends only on `entityRegistry` and a small set of utility imports — no site-builder coupling.

### New `cms` plugin (ServicePlugin)

Add `plugins/cms/`:

- Reads entity schemas at startup via the entity registry
- Generates `admin/config.yml` + `admin/index.html`
- Writes them into the brain's content root (the same directory directory-sync owns)
- Re-generates on `entity:type:registered` events so config stays current
- Idempotent: skips writes when content is unchanged

The plugin doesn't expose tools or HTTP endpoints. It just maintains files on disk.

### Site-builder keeps its current behavior

Site-builder continues to generate `admin/*` into the build output for deployments that have a webserver. It can either:

- Keep its own copy of the generation logic (now using the shared module), or
- Stop generating these files entirely and rely on the cms plugin's content-repo files being copied through

Recommended: site-builder keeps generating into the build output (independent path) and just imports the shared module. No coordination required.

### Rover preset wiring

- `core` preset gets `cms` added (alongside `directory-sync`, `mcp`, `discord`, etc.)
- `default` and `full` presets also get `cms` (along with the existing `site-builder` + `webserver` — site-builder's CMS files in the build output are still served by the webserver and remain the primary CMS surface for those deployments)

If `cms` and `site-builder` both run, the cms plugin writes to the content repo and site-builder writes to the build dist. They don't conflict.

### Operator flow on core

```bash
# One-time:
# 1. Operator visits the hosted Sveltia URL (e.g. https://cms.rizom.ai/<owner>/<repo>)
#    or any Sveltia-compatible UI pointed at their content repo
# 2. They authenticate with GitHub (Sveltia handles this client-side)
# 3. They get a CMS UI driven by the auto-generated admin/config.yml

# Steady state:
# - Brain runs core preset — no webserver, no site-builder
# - Brain writes admin/config.yml + admin/index.html into content root
# - Directory-sync commits those files to the content repo
# - Operator edits content via Sveltia
# - Sveltia commits markdown changes to the repo
# - Directory-sync pulls them in
# - Brain reflects edits as entity updates
```

## What changes

### New

- `shared/cms-config/src/index.ts` — `generateCmsConfig`, `generateCmsBootstrap`
- `shared/cms-config/package.json`
- `plugins/cms/src/plugin.ts` — `CmsPlugin` (ServicePlugin)
- `plugins/cms/src/index.ts`
- `plugins/cms/package.json`

### Modified

- `plugins/site-builder/src/lib/cms-config.ts` — re-export from `@brains/cms-config`, or delete and have callers import directly
- `brains/rover/src/index.ts` — add `cms` to all three presets

### Tests

- `shared/cms-config/test/` — unit tests for generator output (golden YAML, schema → field mapping)
- `plugins/cms/test/` — verify plugin writes expected files, regenerates on schema changes, is idempotent

## Open questions

1. **Where should `admin/` live in the content root?** Directly under the brain-data dir (so the git remote has `admin/` at its root), or under a `_cms/` subdir? Sveltia conventions vary.
2. **Does the cms plugin need an opt-out config?** Some operators may want core without writing CMS files. Default-on probably matches the spirit, but a config toggle is cheap.
3. **Hosted Sveltia URL.** Is rizom going to host a Sveltia instance, or do operators just use Sveltia's public hosting / their own deployment? Affects operator docs but not the code.

## Verification

1. New `plugins/cms` builds and tests pass
2. Schema → CMS config output is byte-identical between site-builder's old path and the shared module
3. `preset: core` with cms plugin enabled: starting the brain writes `admin/config.yml` + `admin/index.html` to brain-data
4. Directory-sync picks up the new files and commits them
5. Pointing Sveltia (hosted or local) at the repo's `admin/` URL produces a working editor
6. Editing content via Sveltia, then `directory-sync_sync`, results in the entity being updated in the brain
7. `default`/`full` presets still have site-builder serving `admin/*` from build output — no behavior change for existing deploys

## Risks

- **Schema drift over time** — the generated config encodes schemas as of plugin startup. Schema changes mid-session won't reflect until restart unless the plugin subscribes to registry events. The design includes that subscription, but it's a non-trivial path to test.
- **Content-repo pollution** — operators who don't want CMS files in their repo will see them appear after upgrade. Default-on is a behavior change. Either ship the toggle from day one or default-off in core (CLI command opts in).
- **CMS auth assumptions** — Sveltia's GitHub OAuth flow assumes a known callback URL. If the operator's content repo is private, the Sveltia instance they use must support the necessary OAuth scopes. Document the requirement.
