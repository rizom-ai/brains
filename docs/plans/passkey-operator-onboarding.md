# Plan: Passkey Operator Onboarding

## Status

Partial. `shell/auth-service` ships the first-passkey setup URL, the anchor-visible retrieval tool, and the auth-service tests below. Remaining work: the generic operator notification contract, the auth-service-plugin bridge, and Discord DM delivery to configured `discord:*` anchors.

## Context

`shell/auth-service` owns the OAuth/passkey bootstrap ceremony. When no operator passkey exists, it generates a one-shot `/setup?token=...` URL. That URL is a bootstrap capability: it must be generated by the brain, remain single-use, and become invalid once the first passkey is registered.

Operator-facing interfaces share the Brain tool surface, but hosted onboarding also needs a system-initiated notification path for trusted operators. Discord is the first delivery target, but auth-service must not depend on Discord or any other interface internals.

## Goals

1. Keep and harden the first-passkey setup flow.
2. Expose setup state through an anchor-visible retrieval tool that any interface can use.
3. Add a generic operator notification path for sensitive onboarding messages, starting with Discord DMs to configured anchors.

## Non-goals

- Do not add an environment variable for the setup token.
- Do not make auth-service depend on Discord, email, or interface internals.
- Do not send setup URLs to public channels.
- Do not infer secret delivery from `allowedChannels`.
- Do not require GitHub Actions logs, hosted log scraping, SSH, webhooks, or doc-brain deployment.
- Do not reintroduce `plugins.mcp.authToken`.

## What ships today

`shell/auth-service` owns the bootstrap ceremony: generates a single-use setup token when no passkey exists, serves `/setup?token=...` and the registration endpoints against that token, clears the token after first successful passkey registration, and refuses further setup once any passkey is registered. `AuthService#getSetupUrl()` / `getOperatorSetupRequired()` expose state, and `auth-service-plugin` registers an anchor-visible `get_passkey_setup_url` tool returning `setup_required` / `complete` / `unavailable`. Setup-URL generation, registration token enforcement, retrieval tool results, and setup-closes-after-registration are all covered by tests.

## Remaining work

### Operator notification contract

Introduce a system notification/request with a shape like:

```ts
interface OperatorNotification {
  audience: "anchors";
  sensitivity: "secret" | "normal";
  title: string;
  body: string;
  actionUrl?: string;
  dedupeKey?: string;
}
```

Exact naming can follow existing messaging conventions. The key design point is that this is a shell/plugin notification contract, not an auth-service-to-Discord call.

Responsibility split:

```text
AuthService
  owns setup token/state and exposes "operator setup required"

AuthServicePlugin
  bridges setup-required state into an operator notification request

Operator notification layer / message bus
  defines audience and delivery semantics

DiscordInterface
  owns Discord transport and sends Discord DMs
```

### Auth-service-plugin bridge

The plugin should subscribe to "setup required" state from `AuthService` and emit an `OperatorNotification` request through the operator-notification layer when the brain boots with no passkey configured. Idempotent: re-emits should dedupe through the notification layer's `dedupeKey`.

### Remaining tests (interface + plugin)

- auth-service-plugin emits/requests operator notification when setup is required
- no notification is requested when a passkey already exists
- Discord does not post token-bearing output into unsafe public/server channels
- Discord resolves `discord:*` anchors and sends DMs
- Discord queues or safely handles notification before client readiness
- Discord delivery failure logs do not include the setup URL

## Interface behavior

Interfaces are responsible for deciding whether their current response or notification channel is safe for a token-bearing URL.

Examples:

- Discord may allow anchor retrieval, but should avoid posting the setup URL in public/server channels. It should DM configured `discord:*` anchors for push delivery.
- CLI can print the URL locally for an anchor caller.
- MCP stdio can return the URL to the local anchor client.
- MCP HTTP may not be usable before OAuth setup; that is a transport limitation, not a tool rule.

The auth-service tool itself should enforce anchor visibility through the existing tool permission system. Additional defense-in-depth checks are acceptable if the tool context exposes enough permission data, but the core rule is interface-agnostic: any anchor may retrieve setup state.

## Discord delivery

Discord delivery should use configured anchors as trusted operators:

- Resolve configured `discord:*` anchors.
- DM the same setup URL to each Discord anchor.
- First successful passkey registration wins.
- Later clicks see setup already completed or invalid token.
- Queue the notification until the Discord client is connected.
- Immediately before sending a secret setup URL, avoid sending if setup has already completed or the URL is no longer valid.
- Log delivery failures without including the setup URL.
- Do not fall back to a public channel by default.

If the current permission service can check anchor status but cannot list anchors, add a narrow read API such as:

```ts
context.permissions.listAnchors("discord"); // -> string[]
```

or an equivalent shell-level recipient resolver. Discord should not scrape raw app config if a cleaner permission/recipient API is available.

Suggested DM copy:

```text
Passkey setup required: https://domain/setup?token=...

This setup link is single-use. The first configured anchor to register a passkey completes operator setup and closes the link.
```

## Local/dev logging fallback

The implementation may keep full setup URL logging as a local/dev fallback for loopback issuers.

For hosted issuers, prefer logging a non-sensitive message rather than the full token-bearing URL. Hosted onboarding docs should prefer the anchor-visible setup retrieval tool or configured operator notifications.

Do not persist the setup token only for delivery unless a later implementation explicitly designs storage, expiry, file permissions, and cleanup semantics. In-memory setup state plus `getSetupUrl()` is preferred.

## Docs

Update onboarding docs to state:

- The setup token is generated by Brain, not provided as env.
- Hosted setup receives the generated URL through configured operator notification or anchor-visible retrieval.
- Initial Discord delivery DMs configured `discord:*` anchors.
- The same URL may be sent to multiple trusted anchors; it is single-use and first registration wins.
- Logs are local/dev fallback only.
