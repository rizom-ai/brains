# Site Packages — Unified Theme + Layout

## Problem

Currently, switching a brain instance's visual identity requires:

1. Editing the brain's source code (rover/index.ts) to swap layout imports
2. Overriding `plugins.site-builder.themeCSS` in brain.yaml
3. Knowing which layout pairs with which theme

Theme, layout, routes, and the site plugin are separate concerns spread across `shared/theme-*`, `layouts/*`, and the brain definition — but they're tightly coupled in practice.

## Solution

A **site package** that bundles everything the site-builder needs into one export:

- Theme CSS
- Layout component
- Routes
- Site plugin (templates, datasources, profile schema extensions)
- Entity route config (what posts/decks are called)

Configured via a single top-level `site` key in brain.yaml.

## Package Structure

```
sites/
  mylittlephoney/
    src/
      index.ts       # default export: SitePackage
      theme.css      # visual identity (moved from shared/theme-mylittlephoney)
    package.json     # depends on @brains/layout-personal
  yeehaa/
    src/
      index.ts
      theme.css      # moved from shared/theme-yeehaa or theme-brutalist
    package.json     # depends on @brains/layout-professional
```

### Site Package Interface

```ts
// shell/app/src/site-package.ts
export interface SitePackage {
  /** Composed theme CSS string (base + theme-specific) */
  theme: string;
  /** Default page layout component */
  layout: ComponentType;
  /** Minimal layout component (optional) */
  minimalLayout?: ComponentType;
  /** Route definitions */
  routes: Route[];
  /** Site plugin factory (registers templates, datasources, schemas) */
  plugin: (config?: Record<string, unknown>) => Plugin;
  /** Entity route config (what things are called, URL patterns) */
  entityRouteConfig: Record<string, { label: string; pluralName?: string }>;
}
```

### Example: `sites/mylittlephoney/src/index.ts`

```ts
import {
  personalSitePlugin,
  PersonalLayout,
  routes,
} from "@brains/layout-personal";
import { composeTheme } from "@brains/theme-base";
import themeCSS from "./theme.css" with { type: "text" };
import type { SitePackage } from "@brains/app";

const site: SitePackage = {
  theme: composeTheme(themeCSS),
  layout: PersonalLayout,
  routes,
  plugin: personalSitePlugin,
  entityRouteConfig: {
    post: { label: "Post" },
  },
};

export default site;
```

### Example: `sites/yeehaa/src/index.ts`

```ts
import {
  professionalSitePlugin,
  ProfessionalLayout,
  routes,
} from "@brains/layout-professional";
import { composeTheme } from "@brains/theme-base";
import themeCSS from "./theme.css" with { type: "text" };

const site: SitePackage = {

```

## brain.yaml Configuration

```yaml
brain: "@brains/rover"
site: "@brains/site-mylittlephoney"

plugins:
  site-builder:
    # themeCSS, routes, layouts are now set automatically from site package
    # only non-site overrides go here
    cms: {}
```

The `site` key is:

- **Top-level**, like `brain`
- **Optional** — rover provides a default site (yeehaa/professional)
- Resolved as a `@`-prefixed package ref (same mechanism as `themeCSS` today)

## Resolution Flow

### 1. Instance Overrides Schema

Add `site` to `instanceOverridesSchema`:

```ts
const instanceOverridesSchema = z.object({
  brain: z.string().optional(),
  site: z.string().optional(), // ← NEW
  // ... rest unchanged
});
```

### 2. Brain Definition

Add optional `site` to `BrainDefinition` for the default:

```ts
export interface BrainDefinition {
  name: string;
  version: string;
  site?: SitePackage; // ← NEW: default site
  capabilities: PluginEntry[];
  // ... rest unchanged
}
```

### 3. Brain Resolver

In `resolve()`, unpack the site package into the right places:

```ts
export function resolve(definition, env, overrides) {
  // Resolve site: yaml override > brain default
  const site: SitePackage | undefined = overrides?.site
    ? getPackage(overrides.site) // resolved from package registry
    : definition.site; // brain's default

  // If we have a site package, inject into capabilities
  if (site) {
    // Auto-register the site plugin
    capabilities.push(
      site.plugin({ entityRouteConfig: site.entityRouteConfig }),
    );

    // Inject site config into site-builder overrides
    pluginOverrides["site-builder"] = {
      ...pluginOverrides["site-builder"],
      themeCSS: site.theme,
      routes: site.routes,
      layouts: {
        default: site.layout,
        ...(site.minimalLayout && { minimal: site.minimalLayout }),
      },
      entityRouteConfig: site.entityRouteConfig,
    };
  }

  // ... rest of resolve unchanged
}
```

### 4. Rover Cleanup

Rover removes all site-specific imports and uses `site` instead:

```ts
// Before:
import {
  professionalSitePlugin,
  ProfessionalLayout,
  routes,
} from "@brains/layout-professional";
import yeehaaTheme from "@brains/theme-brutalist";

// After:
import yeehaa from "@brains/site-yeehaa";

export default defineBrain({
  name: "rover",
  site: yeehaa, // ← default site
  capabilities: [
    // NO professionalSitePlugin here
    // NO siteBuilderPlugin with routes/layouts/themeCSS
    [siteBuilderPlugin, { cms: {} }], // just non-site config
    // ... other plugins
  ],
});
```

5. **Create `sites/mylittlephoney/`** — move theme-mylittlephoney CSS + personal layout refs
6. ~~**Update rover** — use `site: yeehaa` default, remove hardcoded layout/theme~~ ✅
7. **Update mylittlephoney brain.yaml** — `site: "@brains/site-mylittlephoney"`
8. **Update entrypoint generator** — handle `site` package refs same as `themeCSS`
9. **Remove built-in layouts from site-builder** — site-builder should not own any layouts; all layouts (including `default`, `minimal`, `cta-footer`, `default-cta`) should come from site packages. Site-builder just looks up layout names from its config map. This also removes site-builder's dependency on `@brains/default-site-content`.

## What Stays, What Moves

| Current location                      | Stays | Moves to                             |
| ------------------------------------- | ----- | ------------------------------------ |
| `shared/theme-base/`                  | ✓     | — (shared utility)                   |
| `shared/theme-mylittlephoney/`        | ✗     | `sites/mylittlephoney/src/theme.css` |
| `shared/theme-brutalist/`             | ✗     | `sites/yeehaa/src/theme.css`         |
| `layouts/personal/`                   | ✓     | — (reusable layout)                  |
| `layouts/professional/`               | ✓     | — (reusable layout)                  |
| `shared/theme-default/`               | ✓     | — (fallback)                         |
| Other themes (swiss, editorial, etc.) | ✓     | — (available for future sites)       |

Layouts stay as reusable building blocks. Site packages compose a layout + theme into a complete identity.
