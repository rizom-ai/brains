# Plan: Passkey Setup Bootstrap

## Status

Proposed.

## Context

`shell/auth-service` owns the OAuth/passkey bootstrap ceremony. When no operator passkey exists, it generates a one-shot `/setup?token=...` URL. That URL is a bootstrap capability: it must be generated by the brain, remain single-use, and become invalid once the first passkey is registered.

Operator-facing interfaces already share a Brain tool surface. Although these tools are exposed to external MCP clients, they are also invoked internally by interfaces such as Discord through the in-process agent/tool path. Discord does not need MCP HTTP or stdio to use such a tool.

## Goal

Keep and harden the first-passkey setup flow, and expose the generated setup URL through an anchor-visible auth-service tool that any interface can use when the caller is an anchor.

This makes setup retrieval a permanent operator-pull mechanism. Future push notifications can still be added as convenience, but hosted onboarding does not have to wait for push delivery.

## Non-goals

- Do not add an environment variable for the setup token.
- Do not make auth-service depend on Discord, email, or interface internals.
- Do not require system-initiated push messaging for hosted setup.
- Do not reintroduce `plugins.mcp.authToken`.
- Do not make the setup URL public.
- Do not require a webhook, SSH, hosted log scraping, GitHub Actions logs, or doc-brain deployment.

## Auth-service responsibilities

- Generate the setup token internally when no passkey exists.
- Serve `/setup?token=...` only for the valid generated token.
- Require the valid generated token for passkey registration endpoints.
- Clear/close setup after successful passkey registration.
- Refuse setup once any passkey exists.
- Expose setup state through a small internal API and an anchor-visible tool.

## Setup URL exposure contract

Auth-service should provide a narrow internal API for setup state, for example:

```ts
interface OperatorSetupRequired {
  setupUrl: string;
  expiresAt: number;
}
```

`AuthService#getSetupUrl()` should remain available for code that needs to re-read the active setup URL before returning or delivering it.

Auth-service may also support a callback/hook for future push delivery, but the primary retrieval mechanism in this plan is the tool below.

## Anchor-visible setup retrieval tool

Register a tool from `shell/auth-service`, for example:

```ts
auth_get_passkey_setup_url;
```

Tool visibility:

```ts
visibility: "anchor";
```

The tool should be available to any interface whose caller is an anchor. It must not be limited to Discord.

Expected behavior:

```ts
type PasskeySetupToolResult =
  | { status: "setup_required"; setupUrl: string; expiresAt: number }
  | { status: "complete" }
  | { status: "unavailable"; reason: string };
```

Rules:

- If a passkey already exists, return `status: "complete"`.
- If setup is required and a valid setup URL exists, return `status: "setup_required"` with the URL and expiry.
- If setup is required but no valid setup URL is available, return `status: "unavailable"` with a non-sensitive reason.
- The setup URL remains the same active one-shot URL; do not generate per-interface or per-user tokens for this plan.
- First successful passkey registration wins and closes setup for everyone else.

## Interface behavior

Interfaces are responsible for deciding whether their current response channel is safe for a token-bearing URL.

Examples:

- Discord may allow anchor retrieval, but should avoid posting the setup URL in public/server channels. It can ask the anchor to DM the bot or otherwise use a private response path.
- CLI can print the URL locally for an anchor caller.
- MCP stdio can return the URL to the local anchor client.
- MCP HTTP may not be usable before OAuth setup; that is a transport limitation, not a tool rule.

The auth-service tool itself should enforce anchor visibility through the existing tool permission system. Additional defense-in-depth checks are acceptable if the tool context exposes enough permission data, but the core rule is interface-agnostic: any anchor may retrieve setup state.

## Local/dev logging fallback

The implementation may keep full setup URL logging as a local/dev fallback for loopback issuers.

For hosted issuers, prefer logging a non-sensitive message rather than the full token-bearing URL. Hosted onboarding docs should prefer the anchor-visible setup retrieval tool.

Do not persist the setup token only for delivery unless a later implementation explicitly designs storage, expiry, file permissions, and cleanup semantics. In-memory setup state plus `getSetupUrl()` is preferred.

## Tests

Add or keep focused auth-service tests for:

- setup URL is generated when no passkey exists
- setup retrieval tool returns `setup_required` with URL and expiry when no passkey exists
- setup retrieval tool returns `complete` when a passkey already exists
- `/setup` requires the valid generated token
- registration endpoints require the valid generated token
- setup closes after successful passkey registration, if existing tests allow

Interface-specific tests belong with the relevant interface implementation. For Discord, test that token-bearing output is not posted into unsafe public/server channels if such routing is implemented.
