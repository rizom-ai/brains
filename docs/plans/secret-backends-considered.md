# Secret Backends Considered (and Rejected) for v0.1

## Status

The default secret backend in `brain-cli` is **`none`** — no varlock plugin in the schema, values resolve directly from `process.env` (which in CI comes from GitHub Actions secrets).

Two alternative varlock backends were investigated and rejected for v0.1: **Bitwarden** and **1Password**. Both rejections are documented here so a future contributor doesn't repeat the same investigation cold.

This is a record, not a plan to execute. To resurrect either backend, the corresponding "When to reconsider" trigger must change first.

## Decision

`brain init` defaults to `--backend none`. The schema generator's fallthrough branch still emits a generic `@plugin(@varlock/<name>-plugin)` line for any other `--backend` value, so an operator who wants to wire a third-party varlock plugin can hand-tune the result. **No backend other than `none` ships verified end-to-end.**

The `--push-to` flag for `brain cert:bootstrap` and `brain secrets:push` is restricted to `gh` (GitHub Actions secrets via the `gh` CLI). The `1password` value was removed.

## Rejection 1: Bitwarden

### What we wanted

- Schema scaffolds with `@plugin(@varlock/bitwarden-plugin)` + an init line + a bulk-load directive
- `brain secrets:push --push-to bitwarden` populates a Bitwarden Secrets Manager project
- Deploy workflow runs `varlock load`, which resolves all secrets via the plugin in one round-trip

### Why it doesn't work

**1. The published npm artifact is empty.** `@varlock/bitwarden-plugin@0.0.6` has `version: 0.0.0`, `files: []`, and ships only a `package.json`. `npm pack` (with no version specifier) downloads this 302-byte tarball — no compiled plugin, no types, nothing. Installing the package gets you nothing usable.

**2. Even ignoring the empty publish, the documented API is the wrong shape.** Per <https://varlock.dev/plugins/bitwarden/>, the plugin exposes only per-secret UUID lookups:

```env-spec
# @initBitwarden(accessToken=$BITWARDEN_ACCESS_TOKEN)
KEY=bitwarden("12345678-1234-1234-1234-123456789abc")
```

There is no `bitwardenLoadProject(...)` or any equivalent bulk loader. The operator would have to:

1. manually create each secret in Bitwarden
2. capture each secret's UUID
3. embed every UUID into `.env.schema`

That makes the operator the UUID custodian and inverts the `brain secrets:push` → deploy flow we built around bulk-load.

### When to reconsider Bitwarden

Either of these would unblock a fresh evaluation:

- varlock ships a documented bulk-load helper for Bitwarden (e.g. `bitwardenLoadProject(<project-id>)` analogous to 1Password's `opLoadEnvironment(...)`).
- The operator-friendly path becomes "secrets indexed by name, not UUID" — i.e. an operator can scaffold a schema without knowing UUIDs ahead of time.

## Rejection 2: 1Password

### What we wanted

- Schema scaffolds with `@plugin(@varlock/1password-plugin)` + `@initOp(token=$OP_TOKEN)` + `@setValuesBulk(opLoadVault(brain-<instance>-prod))`
- `brain secrets:push --push-to 1password` writes secrets to the matching 1Password vault
- Deploy workflow runs `varlock load`, which resolves all secrets via the plugin in one round-trip

### Why it doesn't work

**1. `opLoadVault(...)` is fictional.** The real `@varlock/1password-plugin@0.3.3` (which IS published with content — 10.5 MB unpacked, includes the WASM core) ships only **`opLoadEnvironment(<environment-id>)`** as its bulk loader. There is no `opLoadVault`. We confirmed this two ways:

- Reading the shipped README: only `op(op://vault/item/field)` (per-secret) and `opLoadEnvironment(<environment-id>)` (bulk) are documented.
- Running `varlock load` against a schema generated by our `env-schema.ts`. The error was explicit: `🚨 Error(s) encountered in .env.schema — Unknown resolver function: opLoadVault()`.

**2. Even if we fixed the function name, the push primitive is also wrong.** `secrets-push.ts` and `cert-bootstrap.ts` both used `op document create --title NAME` to push secrets as **documents in a vault**. The 1Password plugin only reads:

- **fields of items** in a vault (via per-secret `op(op://vault/item/field)` references), or
- **values inside a "1Password Environment"** (via `opLoadEnvironment(...)` — Environments are a specific 1P SDK feature, not the same as vaults)

Documents are not a primitive the plugin can read. So even if we changed the function name, every secret we'd push via `op document create` would be invisible to the loader.

**3. We never tested the load side end-to-end.** The push side worked (`op document create` succeeded against a real account), so we assumed the rest was wired correctly. It wasn't. Two layers of brokenness, neither caught until we ran `varlock load` against the real schema.

### What it would take to make 1Password work

A real implementation would have to either:

- **Switch the read pattern to per-secret `op()` references**: schema emits `KEY=op(op://brain-<instance>-prod/KEY/password)` for every key, push side switches from `op document create` to `op item create --category=password --title=KEY --vault=<vault> 'password=<value>'`. Each secret becomes a vault item with a `password` field. Operator migration: re-push everything as items. ~2-4 hours of code + tests + manual operator migration.
- **Use 1Password Environments via `opLoadEnvironment(...)`**: requires understanding the Environments feature (newer 1P SDK concept), and the push primitive becomes whatever populates an Environment (probably `op env item set` or the 1P SDK directly, not `op document create`). More complex operator setup, more research required.

Neither was justified for v0.1 given:

- The default `--backend none` works end-to-end (verified)
- Foundation/work/ai brain instances all use the same default — no real demand for 1Password specifically
- The cost (account setup + plugin maintenance + dual code paths) outweighs the benefit (centralized secrets store) for solo operator use

### When to reconsider 1Password

- A real operator (you, or a public user adopting brains) actually wants 1Password specifically — i.e. there's a concrete user need, not a "could be useful someday" abstraction.
- A small fix scope is acceptable: implement the per-secret `op()` reference pattern, ~2-4 hours of code + a re-push migration on the operator side.

## What stays in the codebase

After cleanup:

- `lib/push-target.ts` — `PushTarget` is the union `"gh"` (one element). `normalizePushTarget` accepts `"gh"` and `"github"` aliases.
- `lib/push-secrets.ts` — `pushSecretsToBackend` has a single `case "gh":` branch.
- `lib/env-schema.ts` — `secretBackendPrelude` returns empty for `"none"`, falls through to `# @plugin(@varlock/<name>-plugin)` for anything else. No 1Password-specific branch.
- `commands/cert-bootstrap.ts` — `--push-to gh` works; no 1Password branch.
- `commands/secrets-push.ts` — `--push-to gh` works; no `opToken` resolution.

The bootstrap-section detection in `secrets-push.ts`'s schema parser stays as defensive code: any future varlock plugin that emits a `# ---- secret backend bootstrap ----` section will have its tokens excluded from the push set automatically, without needing a name-based exclusion list.
