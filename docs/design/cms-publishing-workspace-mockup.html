Trust & identity · essay 04

Domain as identity

Two brains that have never met can already talk—but only as strangers. Trust should not depend on a shared secret pasted into two configuration files.

A domain is more than a route. It is a durable public claim: a place where capabilities, history, and accountable ownership can meet.

A leaked token is a stolen key. A domain-bound identity is a name that can answer for itself.

The practical work is not to remove credentials. It is to make credentials subordinate to an identity that survives rotation, migration, and changing infrastructure.

Information architecture

Capability, not collection

Publishing sits under Operations. It never pretends to be another entity type, even though it links back into entity editors.

Visual language

Dispatch ledger, not kanban

Numbered rows express durable order and schedule better than the Dashboard’s former three-lane board.

State boundary

Save is not publish

Publication actions stay in the colophon; the black save bar keeps its exact DB → file → git meaning.

Optional composition

Absence leaves no scar
